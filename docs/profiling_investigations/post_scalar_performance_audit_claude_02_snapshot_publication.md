# Review

## Findings

- [severity=high] [category=production] [confidence=high] `src/terminal_screen_model.cpp:2885-2924`, `src/terminal_screen_model.cpp:7076-7100`, scope `Terminal_screen_model::render_snapshot::append_rows`: snapshot construction visits and materializes every visible row regardless of `request.dirty_rows`. Profile shows `render_snapshot_rows_visited=352,030 = render_snapshot_rows_materialized = 1,498 × 235`, scanning 307M cells while only 14,379 dirty rows were requested. `append_rows` self-time is 10.6 s of the 95 s profile (7.07 ms/call avg, max 13 ms), dominating snapshot work; metadata/viewport/dirty-row stages combined are <30 ms. Dirty ranges feed only downstream cache-reuse, not snapshot pruning. Why it matters: this is the single largest snapshot-publication cost in the profile and scales linearly with grid size — exactly what Nelostie's large grid stresses. Concrete follow-up: introduce a row-level identity/generation tag (retained_line provenance + content_generation already exist; primary active rows need a per-row generation) and gate `append_snapshot_cells_from_row` plus `viewport_row_retained_hyperlink_identity_keys` on `request.dirty_rows ? viewport-rebased rows`. Land a slice that emits an opt-in "incremental cells" snapshot variant guarded by a flag so consumers (`build_terminal_render_frame::cells`) can stage.

- [severity=high] [category=architecture] [confidence=high] `src/terminal_session.cpp:5990-6010`: dirty-coalescing runs AFTER `m_screen_model->render_snapshot(request)` returns, then merges the prior snapshot's `dirty_row_ranges` into the new snapshot. So when `m_render_snapshot_synced_generation < m_render_snapshot_generation`, the expensive full-row materialization has already happened. Profile shows `max_unrendered_snapshot_generations=11` (worst-case 11 stacked snapshots), `snapshots_superseded_before_render=112` cumulative skipped generations, and `dirty_coalescing_attempts=45`. Each of those 112 superseded snapshots paid full `append_rows` cost without ever being rendered. Why it matters: when bursts cause GUI backlog, the producer keeps building full snapshots the consumer will never see — the worst-case scaling regime for Nelostie. Concrete follow-up: move dirty-range union into `make_render_snapshot_request` (merge previous-snapshot's ranges into `request.dirty_rows`) so a future incremental-snapshot path can prune; also evaluate skipping snapshot construction entirely when synced<gen if the previous snapshot's content is still valid for the new request (basis + viewport identity match) — currently we always build, even on geometry/IME-only updates.

- [severity=medium] [category=production] [confidence=high] `src/terminal_screen_model.cpp:2891-2902`, `3060-3086`, `6983-7010`: per visible primary-scrollback row, `materialize_retained_history_record` is called twice per snapshot — once via `primary_backing_row` for cells, once via `viewport_row_retained_hyperlink_identity_keys` for hyperlink identity keys (both go through `retained_history.traversal->resolve_cached_row`, with a self-rebuild retry path on `CACHE_ENTRY_INVALID`). This is inside the `append_rows` hot loop. Why it matters: scrollback rows decompress twice for every visible row on every snapshot, even clean ones; the cache may absorb the cost but each call still allocates `retained_row_record_t` and copies the cell vector. Concrete follow-up: materialize the retained record once per row inside `render_snapshot`'s append loop and pass references to both `append_snapshot_cells_from_row` and the hyperlink-metadata extractor. Measure with the `Terminal_screen_model::primary_backing_row::retained_history_materialize` scope already wired up.

- [severity=medium] [category=consolidation] [confidence=high] `src/terminal_session.cpp:867-895` vs `src/vnm_terminal_surface.cpp:112-142`: `append_dirty_range_rows` and `snapshot_with_coalesced_dirty_rows` exist in two translation units with subtly different signatures (session takes new snapshot by value and stores back; surface takes both by const ref and returns a `shared_ptr`). Identity-space predicates `snapshots_share_row_identity_space` (session) and `same_viewport_row_identity_space` (surface) compare different field sets — session includes `row_origin_generation` (`src/terminal_session.cpp:860`), surface does not (it only compares `grid_size` + viewport mapping). Why it matters: divergence risk. A future row-origin/scrollback eviction that goes through surface-side coalescing but not session-side could silently fold ranges across an invalid identity boundary, leaving stale dirty rows. Concrete follow-up: hoist `snapshot_with_coalesced_dirty_rows` and the identity predicate into `render_snapshot.h` (or a sibling header) and have both call sites use a single implementation; assert in tests that both sites take the same fallback when identity differs.

- [severity=medium] [category=production] [confidence=medium] `src/terminal_session.cpp:4370-4394`, profile evidence: publication cadence is "one publish per backend output command." Profile shows 1500 `process_backend_output_command` calls ? 1498 publishes; in burst buckets there are 4-7 snapshots/100 ms (e.g., bucket 30700-30800 has 6 dirty-rows snapshot calls). Each publish costs ~7.6 ms in `publish_render_snapshot` (self+children). Cumulative `publish_render_snapshot` = 11.4 s / 95 s wall = ~12% of GUI thread. Why it matters: when `process_pending_commands` drains many queued commands sequentially, each produces a full snapshot the GUI may not even sync. The producer can outrun the consumer despite invalidation coalescing. Concrete follow-up: investigate batching publication at the end of `process_pending_commands` (or per drain-deadline slice), accumulating dirty rows into one publication. Risks (selection ordering, transcript capture, synchronized-output release timing) are real — start with a measurement-only diagnostic that counts how many snapshots within a single drain would be candidate for collapse, then propose. The existing `post_scalar_performance_audit_codex_d_dirty_publication.md` plan already lists this as option 5.

- [severity=medium] [category=missing-coverage] [confidence=high] `src/terminal_session.cpp:879-895` (`snapshot_with_coalesced_dirty_rows`) is reachable only when `m_render_snapshot_synced_generation < m_render_snapshot_generation` and a prior latest-content snapshot exists. No test in `tests/render_snapshot/`, `tests/backend_session/`, or `tests/surface_host/` directly exercises this session-side function or the identity-mismatch fallback at line 883-887. The surface-side counterpart is exercised by `test_surface_session_single_drain_coalesces_dirty_rows` and `test_surface_session_snapshot_burst_coalesces_to_latest_render` (`tests/surface_host/surface_host_tests.cpp:1079, 1191`), but those tests are end-to-end and do not isolate the predicate. Why it matters: session-side coalescing handles the case where the surface is multi-publication behind; the 45 fires in this profile are exactly that scenario. A regression here would either lose dirty bits (visual stale) or fold across an identity boundary (visual corruption). Concrete follow-up: add unit tests that drive `Terminal_session` through multiple `publish_render_snapshot` calls without invoking `mark_render_snapshot_synced`, assert the final snapshot's `dirty_row_ranges` union the prior ones, and assert identity-mismatch (grid resize between publishes) collapses to a full repaint range.

- [severity=medium] [category=missing-coverage] [confidence=medium] No assertions on `Terminal_session_profile_stats` counters (`dirty_coalescing_attempts`, `dirty_coalescing_applied`, `snapshots_superseded_before_render`, `snapshots_marked_rendered`, `max_unrendered_snapshot_generations`, `zero_dirty_snapshot_publications`). Profile shows `dirty_coalescing_attempts=45` and `snapshots_superseded_before_render=112` are the primary observability into producer/consumer imbalance. Why it matters: these are the only signals that catch silent regressions where coalescing stops firing (e.g., if `m_latest_content_render_snapshot` is reset incorrectly) or where the GUI falls behind unexpectedly. Concrete follow-up: in `backend_session_tests.cpp`, expand the existing generation-bump tests to also assert profile-counter deltas; add a burst-style test that publishes N snapshots without sync, then asserts `dirty_coalescing_applied` advanced by N-1.

- [severity=low] [category=production] [confidence=medium] `src/terminal_screen_model.cpp:6538-6555`: `dirty_rows()` allocates a `std::vector<int>` from `std::set<int>` on every call. Profile: 1500 calls = 528.6 µs total (352 ns mean), so the absolute cost is tiny, but the const-method contract leaks an allocation into the snapshot-request path. Why it matters: hot-loop hygiene — a span-returning API would let `make_render_snapshot_request` consume rows directly without an intermediate copy, and the same change benefits any future caller. Concrete follow-up: expose `dirty_rows_view()` returning a `std::span<const int>` over a stable contiguous buffer (lazily rebuilt from the set), or accept an output iterator. Keep `dirty_rows()` for tests.

- [severity=low] [category=production] [confidence=high] `src/terminal_screen_model.cpp:7022-7042`: `viewport_dirty_rows` for ALTERNATE buffer returns `dirty_rows` by value (`return dirty_rows;` line 7027), forcing a copy even in the common alternate-buffer case. For PRIMARY it allocates a fresh vector. Concrete follow-up: take dirty_rows by const-ref+span, return a small-vector or write into a caller-provided buffer; minor wins for alternate-screen workloads.

- [severity=low] [category=consolidation] [confidence=medium] `src/terminal_session.cpp:44-51` (`model_result_warrants_render_snapshot`) and `:897-904` (`model_should_publish_render_snapshot`) are layered, but the callers add ad-hoc OR clauses (`render_snapshot_metadata_changed`, `m_visual_bell_active`) at each site (`src/terminal_session.cpp:3704, 4370`). The publication gate is therefore expressed in 3+ places with slightly different inputs. Why it matters: a future bug-fix that should propagate to every publish site (e.g., adding a new metadata-change flag) is easy to miss. Concrete follow-up: collapse the gate into a single `should_publish_render_snapshot(model_result, session_state)` helper.

- [severity=low] [category=test-quality] [confidence=medium] `tests/surface_host/surface_host_tests.cpp:1191-1263` (`test_surface_session_single_drain_coalesces_dirty_rows`) asserts `text_content_rebuilds >= 3` as the proxy for "all skipped dirty rows were rebuilt." A regression that keeps the dirty bits but doubles them (e.g., adds two identical ranges) would still produce =3 rebuilds and pass. Why it matters: the test guards against missed dirty rows but does not pin down the *shape* of the final dirty range set. Concrete follow-up: assert the exact `dirty_row_ranges` against expected (rows 0,1,2 contiguous), and assert `text_content_reused` for the unmodified rows.

## Test Assessment

The render-snapshot construction contract has strong unit coverage in `tests/render_snapshot/render_snapshot_tests.cpp`: dirty-row compaction (line 368-373), viewport-relative dirty translation (line 475-491), viewport_changed full-repaint fallback (line 429-434), retained-line provenance / selection lease validation (line 388-470). `tests/backend_session/backend_session_tests.cpp` covers `render_snapshot_generation` cadence around scroll, selection, blocked/release, and resize transitions (hundreds of assertions). `tests/surface_host/surface_host_tests.cpp` covers the surface-side coalescing end-to-end for burst + single-drain cases.

The biggest gaps for this scope:

1. **No direct test of the session-side `snapshot_with_coalesced_dirty_rows`** — the function only fires on producer-ahead-of-consumer scenarios that are hard to construct via the public session API; tests should fake it by holding off `mark_render_snapshot_synced` across multiple publishes.
2. **No identity-mismatch coalescing fallback test** in either coalescing path — i.e., grid resize or row-origin generation change between two unsynced publishes should produce a full-repaint range, not a stale union.
3. **No `Terminal_session_profile_stats` regression assertions** — `dirty_coalescing_applied`, `snapshots_superseded_before_render`, and `max_unrendered_snapshot_generations` are the only observability into the cadence problem this scope cares about.
4. **No test of SELECTION_DERIVED / GEOMETRY_DERIVED interleaved with CONTENT** — the dual-storage in `m_latest_content_render_snapshot` vs `m_latest_render_snapshot` (`src/terminal_session.cpp:5990-5993`) is non-trivial and its coalescing-source selection depends on `purpose`. Misleading: the existing tests imply correctness via end-to-end rendering, but the cross-purpose coalescing routing is not isolated.
5. **No test of sliced backend output amplification** — `process_pending_commands` (`src/terminal_session.cpp:3112-3128`) can split a single backend command into multiple `process_backend_output_command` invocations, each potentially publishing. There's no test asserting how many snapshots a sliced burst produces.

## Suggested Follow-ups

1. **Validate the architectural hypothesis with a measurement-only patch**: add a per-row "would have skipped" counter inside `append_rows` keyed on `(row not in request.dirty_rows) AND retained_line_provenance.content_generation unchanged since last snapshot`. Confirms upper bound of savings (profile predicts ~96%) before committing to redesign.
2. **Add isolated tests for `snapshot_with_coalesced_dirty_rows`** (both session-side and surface-side) covering: matched identity union, identity mismatch ? full repaint, empty prior dirty set, full prior dirty set. Consolidate the two implementations behind one shared helper to drop drift risk.
3. **Implement single-materialize-per-row** for primary scrollback rows by sharing the `retained_row_record_t` between cells and hyperlink-identity extraction (`src/terminal_screen_model.cpp:6991-7000` + `3060-3086`). Low-risk refactor with measurable wins under the existing `Terminal_screen_model::primary_backing_row::retained_history_materialize` scope.
4. **Add profile-counter assertions** to existing publication tests: e.g., the burst-coalescing surface test should assert `session->profile_stats().dirty_coalescing_applied` increments and `snapshots_superseded_before_render` advances by the burst size - 1.
5. **Investigate publication cadence collapse within a single drain**: instrument a counter at the end of `process_pending_commands` for "number of publish_render_snapshot calls this drain" and feed it through `Terminal_session_profile_stats` (or a new counter) so we can quantify how much per-drain coalescing would save. Use that data to propose a deferred-publication slice — only the last command in a drain publishes, prior dirty rows merged into the request.
6. **Re-profile after the existing audit's option 1 (no-op fast path in `publish_pending_changes`)** lands; that change is ~zero-risk and will let the next profile expose the next bottleneck cleanly without `publish_pending_calls` noise.
