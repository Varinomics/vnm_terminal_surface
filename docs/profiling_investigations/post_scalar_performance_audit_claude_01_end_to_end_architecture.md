## Findings

- [severity=high] [category=production] [confidence=high] `src/terminal_screen_model.cpp:2825-2924` — `render_snapshot()` iterates **all 235 viewport rows** every snapshot regardless of `dirty_row_ranges`, calls `viewport_row_cells(viewport,row)` per row, and pre-reserves `rows*columns ˜ 205,155` cells (line 2834). Profile: `render_snapshot_rows_visited = 352,030` ˜ `1,498 × 235`, `render_snapshot_cells_emitted ˜ 134 M`, average 7.07 ms/snapshot × 1,498 = 10.6 s = 66 % of GUI thread. Why it matters: every snapshot is a "full repaint" payload even when 75/235 rows changed — downstream consumers (frame build, packed pass, text grouping) inherit full-grid cost. Concrete follow-up: emit only `dirty_row_ranges` cells (or a `(row_id ? CellSpan)` map indexed by snapshot row) and have build/QSG paths fall back to the previous frame's cells for clean rows. Quantify the ceiling first by skipping non-dirty rows in `append_rows` behind a flag and re-running Nelostie.

- [severity=high] [category=production] [confidence=high] `src/terminal_screen_model.cpp:7156-7184` and `:6984-7010` — `viewport_row_cells()` returns `std::optional<std::vector<Cell>>` **by value**, and the underlying `primary_backing_row()` already returns `std::optional<Terminal_screen_row>` by value. That is two full `std::vector<Cell>` copies (~42 KB per row at 873 cols × ~48 B `Cell`) per row materialization, plus `append_snapshot_cells_from_row()` calls `row_cells_for_current_geometry()` which can allocate a third `visual_projection` (line 7081-7083). Why it matters: 235 rows × 1,498 snapshots × =2 copies = ˜ 700 K vector copies (˜ 20–30 GB allocator churn over the run) sitting under the 10.6 s `append_rows` cost, and it survives even after `cells.reserve()` because the per-row temporaries are independent. Concrete follow-up: change `viewport_row_cells`/`primary_backing_row` to return `const std::vector<Cell>*` (or a `Cell_span_t`) for the no-projection path; only materialize a vector when geometry projection is actually needed. Verify with `printable_ascii_row_copies` and a heap allocation counter.

- [severity=high] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:6128-6341` (`::cells`) and `:5957-6035` (`::packed_data`) — two complete passes over `snapshot.cells`. Both call `cell_text_decoration_enabled`, `classify_terminal_simple_content_cell`, `snapshot_row_is_dirty`, `style_attributes.attributes(cell.style_id)`, and `block_cursor_covers_cell` for every cell. Profile cumulative: 13.28 s + 12.75 s = 26 s ˜ 66 % of render thread; `frame_cell_pass_input_cells = frame_packed_pass_input_cells = 55,837,192` confirms identical cell counts and overlap. `frame_dirty_row_lookup_count` is exactly `2 × frame_cells_considered` because each pass does `stats.dirty_row_lookup_count += snapshot.cells.size()` unconditionally (`:6074`, `:5970`). Why it matters: the per-cell classification is the dominant cost on the render thread; doing it twice doubles it. Concrete follow-up: merge `::cells` and `::packed_data` into a single pass that produces both `text_runs` and packed spans in one iteration; or, if pipeline separation is required, produce packed data directly from the snapshot and stop emitting `Terminal_render_text_run` for cells that are only consumed by the packed path.

- [severity=high] [category=architecture] [confidence=high] `include/vnm_terminal/internal/render_snapshot.h:427-442` — the snapshot is a **flat `std::vector<Terminal_render_cell>` indexed nowhere**. Downstream every consumer either re-derives row groupings (`qsg_terminal_renderer.cpp:5804-5830 build_explicit_snapshot_row_table`, 235 inner-vector allocations + stable_sort per frame; `:3228 text_run_groups_by_viewport_row`, 450 ms cumulative) or scans linearly with a `snapshot_row_is_dirty` lookup per cell. Why it matters: this is the architectural mismatch under every hot path — the dirty-row metadata is per row, the payload is per cell, and there is no row index, so each layer pays O(cells) to recover row identity. Concrete follow-up: index the snapshot by row at construction (`vector<RowSpan{first_cell, count, content_generation}>`) or split into `vector<RowRecord>` with cells inline; this would let frame-builder skip non-dirty rows entirely and avoid `build_explicit_snapshot_row_table`. This is the same direction as `docs/primary_backing_redesign_plans/flat_ring_phase_3_active_to_retained_record_producer.md` and `phase_5c_retained_lookup_cache_replacement.md` — surface this audit as input to that redesign.

- [severity=medium] [category=production] [confidence=high] `src/terminal_session.cpp:5932-6042` — 1,498 snapshots constructed and `make_shared`-wrapped, but only 662 frames painted (cumulative_renderer_stats). `snapshots_marked_rendered=1,453` - `frames_published=662 = 791` snapshots were "synced to the surface" but replaced by `set_render_snapshot()` (`vnm_terminal_surface.cpp:5366-5372`) before `updatePaintNode` ran. At 7.07 ms each that is ˜ 5.6 s of GUI-thread `render_snapshot` work that never reached the screen. The existing `dirty_coalescing` (line 5994-6010) ran only 45 times. Why it matters: the producer outruns the consumer and over-pays; the GUI thread budget is the same one that ingests output, so wasted snapshots directly delay PTY drain. Concrete follow-up: gate snapshot materialization behind "previous snapshot consumed by render thread" (acknowledgement from `updatePaintNode`), or build a lazy/deferred snapshot that materializes rows on demand from the model. At minimum, track and surface this waste rate as a first-class profile counter alongside `snapshots_superseded_before_render` (which only counted 112 here, undercounting the real waste).

- [severity=medium] [category=production] [confidence=high] `src/terminal_screen_model.cpp:6212-6262` — `mark_dirty()` is called 4.38 M times with 4.26 M duplicates (97 %). - [severity=medium] [category=production] [confidence=high] `src/terminal_screen_model.cpp:6598-6635` — `publish_pending_changes()` is called **174,953 times** for 14,379 published unique rows (12 calls per useful publication). ## Test Assessment

For the audit scope (snapshot ? surface ? frame ? QSG), correctness coverage is strong: `render_snapshot_tests.cpp`, `backend_session_tests.cpp` (13 K LOC), `render_frame`, `qsg_render`, `qsg_text_node`, `screen_operations`, `viewport`, `resource_lifecycle`, `conformance`, `parser_*`, and `unicode_width` suites collectively exercise the structural contracts: provenance preservation, selection metadata, IME overlay handle stability, snapshot-handle identity, alternate-buffer switching, dirty-row viewport relativization, geometry-in-sync metadata, public-projection scroll. Test_session_ime_overlay_does_not_clone_render_snapshot (`backend_session_tests.cpp:735`) is the right pattern — it asserts shared_ptr identity, i.e. "we did not copy."

The major gap is performance regression coverage. None of the suites enforces budgets on:
- cells emitted per snapshot vs dirty cells (would catch the all-rows iteration in `append_rows`);
- row-vector copies per snapshot construction (would catch `viewport_row_cells`'s value-return regression);
- frame-build cell passes (would catch a re-introduction of a 2× pass after a merge);
- ratio of snapshots produced to frames painted (would catch the producer-outruns-consumer regression).

The single most misleading existing signal is `cumulative_renderer_stats.frame_dirty_row_lookup_count`: the name suggests a per-cell dirty check, but in practice it is two unconditional `+= cells.size()` adds — a code-reviewer relying on this metric would conclude (incorrectly) that two lookups per cell is the cost model.

## Suggested Follow-ups

Priority order:

1. **Quantify the ceiling of "dirty-only snapshot"** — gate `render_snapshot::append_rows` (`terminal_screen_model.cpp:2885-2924`) behind `is_dirty_row(row)` and re-run Nelostie. If GUI thread drops from 16 s toward 6–8 s the architectural direction is confirmed; this is the cheapest experiment with the largest signal.
2. **Eliminate row copies in `viewport_row_cells`/`primary_backing_row`** (`:7156, :6984`) by returning `const Terminal_screen_row*` or `Cell_span_t` for the no-geometry-projection path. Measure heap allocations with an allocator hook.
3. **Merge `build_terminal_render_frame::cells` and `::packed_data`** (`qsg_terminal_renderer.cpp:6128, :5957`) into a single pass; the duplicated classification work is the second-largest single saving and is mechanically straightforward.
4. **Add row-indexed snapshot layout** (`render_snapshot.h:427-442`) so frame-build can skip clean rows; align with `flat_ring_phase_3` and `phase_5c` redesign so this is not done twice.
5. **Couple snapshot production to consumer acknowledgement** (`terminal_session.cpp:5932`, `vnm_terminal_surface.cpp:5366`): only publish when the prior snapshot has been consumed by `updatePaintNode`. Drop the existing weak `snapshots_superseded_before_render` counter and replace with `snapshots_consumed_by_paint` plus the difference vs publications.
6. **Add performance budget tests** in `tests/render_snapshot/` and `tests/render_frame/` for the four budgets named in the Test Assessment.
7. **Consolidate prior audits** into a one-page living index in `docs/primary_backing_redesign_plans/README.md` so this finding set stays attached to the redesign in flight.
