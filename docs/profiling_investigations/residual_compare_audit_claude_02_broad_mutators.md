# Review

## Findings

- [severity=high] [category=production] [confidence=high] `src/terminal_screen_model.cpp:4862-4894` (`install_cell_span`) takes `const std::vector<Cell> before_cells = screen_row.cells;` for every non-ASCII spacing scalar. This is the residual broad full-row snapshot on the printable path: profile span_local shows `row_content_generation_comparisons=887,639` with `row_content_generation_comparison_cells=772,753,829` (˜870 cells/comparison ˜ full row width 873). The mutation footprint is bounded to the union of (the original wide span containing `position`, computed via `cell_base_position`) and `[position.column, position.column + display_width)`. **Safety:** local detection is sound if it widens to the original span base/width before `clear_cell_at` plus the new write range. Follow-up: introduce `spacing_scalar_span_changes_selection_content(row, base_position, original_display_width, new_display_width, new_text)` analogous to `printable_ascii_span_changes_selection_content` at `:4274`, gated by an idempotency test matrix (no-op rewrite of identical CJK glyph, style-only rewrite, hyperlink-only rewrite).

- [severity=high] [category=production] [confidence=high] `src/terminal_screen_model.cpp:4978-4998` (`erase_row_range`) snapshots the entire row before erasing a possibly tiny range (`[first_column, last_column]`). Called by `clear_screen_before_cursor` (`:5000`), `clear_screen_after_cursor` (`:5008`), `erase_in_line` (`:5113-5115`), and `erase_characters` (`:5126`). A single ED 0 from row 0 produces 235 full-row copies + 235 full-row scans on a 235-row terminal — a multiplicative hot path matching the residual 887K comparisons. **Safety:** the modified footprint is `[base(first_column), base+width(first_column)) ? [first_column..last_column] ? [base(last_column), base+width(last_column))`; a local scan that walks that widened range and compares to `erased_cell()` is sound. Boolean known-change is *not* sufficient here because the existing test `erase-character on already blank content does not increment generation` (`tests/.../model_ops_tests.cpp:3140`) relies on idempotency detection. Follow-up: extract `erased_range_changes_selection_content(row, widened_first, widened_last)` with an explicit wide-span-widening contract, then drop the `before_cells` copy.

- [severity=high] [category=production] [confidence=high] `src/terminal_screen_model.cpp:5016-5034` (`erase_visible_screen`) takes per-row `before_cells` copies for all 235 rows before `fill_row_with_erased_cells`. Measured: `mark_all_dirty_calls=87` over the session (span_local profile line 24) ? up to ~20K of the 887K residual comparisons originate here. The after-state is uniform `erased_cell()`, so the before-snapshot is redundant. **Safety:** a single-pass scan-without-copy (`for cell in row.cells: if !cells_have_same_selection_content(cell, erased_template) ? changed`) yields identical semantics with zero allocation. Boolean known-change is *not* safe: ED 2 on an already-blank screen must not increment generation (no explicit test today, but consumers in `qsg_terminal_renderer.cpp:3349-3352` and `terminal_session.cpp:4880-4882` would re-tessellate / drop selection unnecessarily). Follow-up: collapse the `before_cells` snapshot into an inline cell-by-cell scan and add a regression test for ED 2 idempotency.

- [severity=medium] [category=production] [confidence=high] `src/terminal_screen_model.cpp:5132-5205` (`insert_cells` at `:5138`, `delete_cells` at `:5173`) snapshot the entire row but only mutate `[cursor_column, columns)` plus the wide-boundary widening at `cursor_column` and `cursor_column + count` (resp. `columns - count`). **Safety:** local detection is sound by scanning the mutated range only; the existing tests at `model_ops_tests.cpp:3129-3135` (`insert/delete-cell content shift increments generation`) cover content-changing cases but no test currently asserts the idempotent shift case (insert/delete on an already-blank tail), so a stricter local check can be introduced together with that missing test. Follow-up: introduce `shift_range_changes_selection_content(row, first, last)` and add a no-op shift regression covering ICH on a row of erased cells.

- [severity=medium] [category=missing-coverage] [confidence=high] `tests/screen_operations/model_ops_tests.cpp:3102-3208` (`test_retained_line_content_generation_mutations`) does not exercise idempotency for: ED 2 (`erase_visible_screen`) on an already-blank screen, ED 0/1 (`clear_screen_after_cursor`/`clear_screen_before_cursor`) on already-blank ranges, ICH (`insert_cells`) with zero non-erased cells in the shifted source, and DCH (`delete_cells`) with zero non-erased cells in the shifted source. The current tests assert change-cases for these (e.g. `:3130-3135`) but not the no-op cases that the local-detection conversion must preserve. Why it matters: these are the exact correctness gates the conversion needs to land safely. Follow-up: add four idempotency assertions before any mutator conversion lands; gate each conversion slice on its corresponding new assertion.

- [severity=medium] [category=missing-coverage] [confidence=medium] `tests/screen_operations/model_ops_tests.cpp:3102-3208` lacks generation idempotency for CJK/wide spacing scalar rewrites that hit `install_cell_span`. The narrow case `wide-cell occupancy change increments generation` (`:3156`) covers the change-direction but there is no assertion that re-issuing the identical wide glyph at the same column leaves the generation untouched. With `install_cell_span` rewritten to use local detection, this becomes the load-bearing correctness test. Follow-up: add `idempotent wide-cell rewrite does not increment generation` and `style-only rewrite of identical wide cell does not increment generation` before converting `install_cell_span`.

- [severity=low] [category=redundancy] [confidence=high] `src/terminal_screen_model.cpp:4162-4187` (`resize_rows`) copies the entire row before resize + repair. Resize is rare (per `mark_all_dirty_calls=87` over the whole session, of which the resize subset is smaller), so the full-row copy is harmless but unnecessary. **Safety:** if `grid_size.columns` changed, content is always considered changed; if columns are equal and only row count grew/shrank, generation does not need to advance for retained rows present in both ranges (no per-cell content delta). Boolean known-change keyed off `(old_columns != new_columns) || repair_wide_spans_touched_anything` suffices. Follow-up: low-priority cleanup; can land alongside the broader conversion or be deferred indefinitely.

- [severity=low] [category=redundancy] [confidence=high] `src/terminal_screen_model.cpp:4798-4860` (`append_zero_width_scalar` autowrap branch, line 4835-4837) copies the whole row to detect a single-cell `clear_cell_at(target)` mutation. The footprint is exactly the wide span at `target` (base + continuations). Local detection trivially captures it. This branch is rare (combining sequence appended when cursor sits past column 0 with autowrap pending), so the win is tiny but the conversion is one-line. Follow-up: replace with `cells_have_same_selection_content` on the affected base cell + its continuations.

- [severity=low] [category=consolidation] [confidence=medium] `src/terminal_screen_model.cpp:4252-4290` already establishes the proven local-detection pattern (`printable_ascii_cell_changes_selection_content`, `printable_ascii_span_changes_selection_content`) that drove the span-local removal of `printable_ascii_row_copies`. The remaining six broad mutators each warrant a parallel `<operation>_changes_selection_content` helper with a documented widening rule for wide-span side effects, rather than ad-hoc inline scans. Follow-up: define the helper family as part of the first conversion slice (recommend `install_cell_span` first) and reuse for subsequent conversions.

## Test Assessment

The change-direction coverage in `test_retained_line_content_generation_mutations` (`tests/screen_operations/model_ops_tests.cpp:3102-3208`) is solid: it spans printable ASCII, ICH, DCH, ECH (idempotent + changing), wide-cell occupancy, ASCII-over-wide-continuation, combining/variation sequences, style-only rewrite, hyperlink-only rewrite. The dirty-rows and snapshot validation in `test_erase_operations_and_wide_damage`, `test_blank_fill_operations_use_current_style`, and `test_insert_delete_lines_cells_and_tabs` exercise wide-span widening for `erase_row_range`, `install_cell_span`, `insert_cells`, and `delete_cells`, which the conversion needs to preserve.

The most important gaps are idempotency assertions for the broad mutators themselves: there is no test verifying that ED 2 on an already-blank screen, ED 0/1 on already-blank tail/head ranges, ICH on a row of erased cells, DCH on a row of erased cells, identical CJK rewrite, or style-only wide-cell rewrite all leave `content_generation` untouched. These are precisely the no-op cases the conversion's local detection must continue to honor; without them the conversion would silently regress to "always advance" without the test suite noticing. The misleading test is the absence of a no-op shift test for ICH/DCH — current passes would still pass if a future change advanced generation on every shift.

## Suggested Follow-ups

1. Land the four idempotency assertions called out above (ED 2, ED 0/1, ICH no-op shift, DCH no-op shift) plus the two wide-glyph idempotency assertions, before any mutator conversion. These form the safety gate.
2. Convert `install_cell_span` first (highest-frequency residual; matches the span-local model directly) by introducing `spacing_scalar_span_changes_selection_content` with explicit wide-span widening. Re-run the `nelostie_profile_span_local` capture and verify `row_content_generation_comparisons` drops materially while `printable_ascii_local_cells_inspected` rises proportionally.
3. Convert `erase_row_range` second (highest per-call multiplicity via clear_screen_before/after_cursor). Use a single helper `erased_range_changes_selection_content` and gate on the EL/ED idempotency tests from step 1.
4. Convert `erase_visible_screen` third by inlining a no-copy scan against `erased_cell()`. Verify in the profile that the per-bucket comparison cells drop when `mark_all_dirty` buckets fire.
5. Convert `insert_cells`/`delete_cells` fourth using a shared `shift_range_changes_selection_content` helper.
6. Treat `resize_rows` and `append_zero_width_scalar` as optional cleanups — low ROI; bundle only if the helper family lands cleanly.
7. After each slice, refresh `nelostie_profile_span_local.txt` and add the new `row_content_generation_comparisons` / `row_content_generation_comparison_cells` deltas to the phase plan as the validation gate. Treat any regression in `row_content_generation_advances` count vs. the corresponding hardened baseline as a correctness flag for the conversion.
