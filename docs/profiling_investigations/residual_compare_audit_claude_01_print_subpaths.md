## Findings

- **[severity=high] [category=production] [confidence=high]** `src/terminal_screen_model.cpp:4862-4896` (`install_cell_span`) — every non-printable-ASCII scalar funnels here via `put_text ? put_scalar ? put_spacing_scalar ? place_cell_text ? install_cell_span` (4562-4570, 4898-4909) and via `append_zero_width_scalar ? install_cell_span` (4853). The function unconditionally captures `const std::vector<Cell> before_cells = screen_row.cells;` (4872) and finishes with `advance_row_content_generation_if_changed(screen_row, before_cells)` (4894). The mutation actually touches `[position.column, position.column + display_width - 1]` plus the at-most-one wide base that `clear_cell_at(position)` (4873, dispatched via `cell_base_position` 5253-5267) may overwrite — a window of size O(1)..O(display_width). Why it matters: this is the dominant residual under `apply_action::print_text` (876,318 / 887,639 compares; 4.58 s wall in the span_local profile). Concrete follow-up: mirror the printable ASCII pattern — accumulate `selection_content_changed` over (a) the original base cell at `cell_base_position(position)` and its continuation cells, (b) the new install range `[position.column, position.column + display_width)` — and call `advance_row_content_generation_with_change_flag` (already declared at 4314). Leave `mark_terminal_content_changed` (4869) and `mark_dirty` (4895) untouched so primary-repaint-recovery ambiguity tracking is preserved.

- **[severity=high] [category=production] [confidence=high]** `src/terminal_screen_model.cpp:4830-4850` (`append_zero_width_scalar` autowrap branch) — when a combining mark widens an existing cell past the right margin and autowrap is on, the branch performs `before_cells = screen_row.cells; clear_cell_at(target); advance_row_content_generation_if_changed(screen_row, before_cells);` (4835-4837) **before** wrapping and then calls `install_cell_span` on the destination row — so a single combining-mark wrap fires **two** full-row compares (4837 + 4894). Why it matters: low call count relative to install_cell_span, but the compounded compares make it the highest cost-per-event residual under print_text. Concrete follow-up: the source row mutation here is a single `clear_cell_at(target)`. The local window is `cell_base_position(target)..base+display_width-1`. Detect with the same span-local helper as the install path, then call `advance_row_content_generation_with_change_flag(screen_row, flag)`. The subsequent `install_cell_span` will be cheap once finding 1 is in place.

- **[severity=medium] [category=production] [confidence=high]** `src/terminal_screen_model.cpp:4252-4290` (`printable_ascii_cell_changes_selection_content` / `printable_ascii_span_changes_selection_content`) — these are the existing span-local detectors built for printable ASCII. They construct an `intended_cell` with `display_width=1, wide_continuation=false, occupied=true` and compare only the four selection-content fields. For the wide / spacing scalar path we need analogous helpers that build an intended *base + continuation* pair from `(text, display_width)`; for the combining path the intended cell carries `cell.text + combining_text` and the new measured width. Why it matters: without a shared helper, install_cell_span and append_zero_width_scalar would each grow ad-hoc local detectors that risk diverging from `cells_have_same_selection_content` (4225-4233). Concrete follow-up: add `spacing_scalar_cell_changes_selection_content(row, position, text, display_width)` (or equivalent) next to the existing two helpers, returning bool. Reuse `cells_have_same_selection_content` under the hood; do not duplicate the field list.

- **[severity=medium] [category=production] [confidence=medium]** `src/terminal_screen_model.cpp:4923-4939` (`clear_cell_at`) calls `cell_base_position` which walks left to find a wide base (5256-5266). The span-local detector for `install_cell_span` must therefore inspect the same potential base column as the existing implementation does *before* mutation — otherwise a "spacing scalar overwriting a wide continuation" event could be misclassified as unchanged. Why it matters: this is the corner case the existing test at `tests/screen_operations/model_ops_tests.cpp:3159-3165` catches today for the ASCII path; the non-ASCII path has no equivalent gate. Concrete follow-up: in the install_cell_span span-local helper, call `cell_base_position(position)` *before* `clear_cell_at`, then compare the existing base cell (plus any continuation columns inside `[position.column, position.column + display_width)`) against the intended layout. Capture this in a dedicated unit test before landing.

## Test Assessment

`tests/screen_operations/model_ops_tests.cpp:3102-3207` (`test_retained_line_content_generation_mutations`) is strong for the ASCII path and most existing producers — it covers printable span bump (3110-3114), idempotent rewrite (3118-3120), insert/delete-cell bumps (3127-3135), erase character bump and idempotency (3137-3150), wide-cell occupancy change (3152-3157), ASCII-over-wide-continuation (3159-3165), no-autowrap clip identity and last-cell change (3167-3175), single combining bump (3177-3182), variation sequence change (3184-3189), style-only no-bump (3191-3196), hyperlink-only no-bump (3198-3205).

Most important missing coverage for the *next* slice (install_cell_span / append_zero_width_scalar autowrap):

- **Idempotent wide rewrite**: write `e4b880` (?), then `\r` + `e4b880` ? expect *no* generation bump. Today the full-row compare catches this; a span-local detector must too. Not directly tested — the existing wide test (3152-3157) asserts the *change* direction, not the idempotent direction.
- **Idempotent spacing scalar rewrite**: same shape with any non-ASCII scalar where text + display_width + occupied + wide_continuation are unchanged.
- **`append_zero_width_scalar` autowrap branch (line 4830-4850)**: a base cell at the right margin combined with a combining mark that bumps measured width past the remaining columns, with autowrap on, in a grid narrow enough to force the branch. No existing test exercises that code path; the line 4837 compare is currently unverified by tests.
- **Style-only rewrite on a wide cell**: symmetric to ASCII style-only at 3191-3196 — write `?` with default style, then `\r` + `\x1b[31m` + `?` ? expect *no* bump. Asserts `install_cell_span` excludes `style_id` from selection-content identity.
- **Hyperlink-only rewrite on a wide cell**: symmetric to ASCII hyperlink-only at 3198-3205 — verifies `install_cell_span` excludes `hyperlink_id`.
- **Span-local-vs-row-snapshot parity oracle for non-ASCII**: drive a varied UTF-8 stream (ASCII + CJK + combining + variation selectors + style/hyperlink changes) through two model instances — one using the existing `advance_row_content_generation_if_changed` path, one using the proposed span-local helper — and assert per-row `content_generation` sequences match step-by-step. The same prior-audit recommendation (`content_generation_copy_compare_audit_claude.md:97`) applies one tier deeper.

The existing tests run on 1-row, 4-8 column grids, so they will pass identically pre- and post-conversion regardless of compare frequency. They cannot prevent regressions in the *frequency* of bumps; pair the parity oracle with the existing counters (`row_content_generation_advances`) and assert counter equality between the two paths.
