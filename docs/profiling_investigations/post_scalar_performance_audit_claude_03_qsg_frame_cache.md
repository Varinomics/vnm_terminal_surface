## Findings

- [severity=high] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:5957-6036` and `:6128-6336` Ś `build_terminal_render_frame` performs two near-identical full-cell traversals (`::cells` 13.28 s + `::packed_data` 12.75 s; 661 frames, ~40 ms combined per frame, ~80 % of `build_terminal_render_frame`). Both call `classify_terminal_simple_content_cell`, `snapshot_row_is_dirty`, and resolve style attributes for every cell, and the packed pass additionally rebuilds an explicit per-row table via `build_explicit_snapshot_row_table` (5804-5830) and `stable_sort` (even though `Terminal_screen_model::render_snapshot::append_rows` already appends cells in row-major, column-ordered fashion Ś see `src/terminal_screen_model.cpp:2885-2924`). The duplicated dirty/style work scales with `cells ū frames` and dominates frame construction at Nelostie sizes. Why it matters: ~26 s of the 39.6 s `updatePaintNode` budget is paid in this duplication, leaving little room for downstream optimization. Concrete follow-up: merge packed-text/graphic emission into the main cell pass, computing classification + style + (cell rect, packed span, text_run, decoration, cursor-overlap exclusion) in a single visit; reuse the row-major iteration the model already provides instead of re-bucketing.

- [severity=high] [category=redundancy] [confidence=high] `src/qsg_terminal_renderer.cpp:5837-5890`, `:5957-6036` (write); `:2090-2148`, `tests/qsg_render/qsg_render_tests.cpp:3843-3895`, `tests/render_frame/render_frame_tests.cpp:1087-1141` (the only readers) Ś `frame.packed_text_spans` and `frame.packed_text_bytes` are populated for every frame but **never consumed by any renderer code path**. `packed_hard_graphic_rects` reads only `packed_rows`, `packed_graphic_spans`, and `packed_graphic_codepoints`. `test_qsg_packed_sidecars_do_not_affect_visual_output` (3843) and `tests/qsg_render/.../qsg_render_tests.cpp:1764` actively clear these fields and confirm pixel parity Ś i.e. the test contract is that they are sidecars. They still cost per-cell UTF-8 encoding (`text.toUtf8()` on every fast-text-eligible cell) and `vector<char>::insert` calls inside the per-frame hot path. Why it matters: half of the packed_data 12.75 s Ś roughly **6 ms/frame** Ś is producing renderer-unused data. Concrete follow-up: either delete the text-span emission and reserve packed-data work for graphics only, or gate it behind a `purpose`/transcript flag so the render thread doesn't pay for it on every frame.

- [severity=high] [category=redundancy] [confidence=high] `src/qsg_terminal_renderer.cpp:3186-3213`, `:3595-3611` Ś `text_resource_key` is built for every dirty/changed row (7,906 cumulative, 107.8 ms total), but the cumulative `text_key_match_reuses=0` (renderer stats line 880) means the key has **never matched a cached key in 662 frames**. The descriptor-reuse path (`text_resource_row_descriptor` comparison at 3545-3561) catches all 1,758 identical-content reuses; the byte-serialized key adds no recovery beyond it. The key also bloats `text_key_bytes` and forces per-byte UTF-16 copies into `QByteArray`. Why it matters: ~100 ms of cumulative cost on a path that has zero observed value, plus extra heap traffic. Concrete follow-up: delete the key build/compare in the dirty-rebuild path (lines 3595-3611) and rely on the descriptor; or, if the key is meant as a safety net, instrument the failure paths and confirm whether descriptor coverage is intentionally narrower than key coverage.

- [severity=high] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:3355-3387`, `:3525-3527` Ś `text_resource_row_descriptor(frame, group)` runs **eagerly for every considered group** (93,517 calls, 1.06 s total = mean 11.3 Ąs). Of those, 83,853 (~90 %) enter the `clean_cache_skip` branch, which only checks `text_resource_descriptor.has_value()` and never reads the contents; only 1,758 calls (~1.9 %) actually drive a content compare in `text_resource_descriptor_reuse`. The descriptor build itself reserves a `vector<Text_resource_row_run_descriptor>`, allocates per-run QString copies, and computes `row_local_rect` for every run in the row, all to satisfy a `has_value()` predicate on the clean-cache path. It also walks `row_has_cursor_text_run` / `row_has_preedit_caret` (linear scans of `cursor_text_runs` / `decorations`) per row. Why it matters: ~1 s/run cumulative, plus heap pressure from QString duplication, all to gate paths that don't need the contents. Concrete follow-up: short-circuit the eager call Ś compute the descriptor only on the dirty-or-stale branch where the content compare is actually invoked, and use a cheap boolean predicate (cursor_overlap, preedit_overlap, any clipped run) for the clean-cache-skip gate.

- [severity=medium] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:5793-5802`, `:6005`, `:6146` Ś `snapshot_row_is_dirty` is a linear scan over `dirty_row_ranges` invoked **per cell, twice per frame** (cumulative `frame_dirty_row_lookup_count=111,674,384` ś 2 ū `cells_considered`). Even though dirty ranges are usually compact, the call is in the innermost hot path of both the cells pass and the packed pass. Why it matters: with 102,975 cells ū 2 passes ū ~3 ranges average, this is a non-trivial constant tax inside the per-cell loop. Concrete follow-up: precompute a `std::vector<bool> row_dirty(grid.rows)` once per snapshot and index it directly; this also lets the packed/cells unification remove one of the two scans entirely.

- [severity=medium] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:3228-3281`, `:3500` Ś `text_run_groups_by_viewport_row` allocates `vector<int>(grid.rows)` and `vector<optional<row_text_group_t>>(grid.rows)` per frame, and walks `frame.text_runs` twice (once for counts, once to populate). Total 450.9 ms / 662 calls (0.68 ms mean). With Nelostie's 235 rows and ~50 k runs/frame, the optional<>-of-vector grid is a notable allocation source. Why it matters: contributes to GC pressure on the render thread and adds a fixed 0.68 ms baseline per frame regardless of dirty count. Concrete follow-up: keep a renderer-owned scratch buffer (reset, not destroyed, between frames), or Ś better Ś drive group construction directly from a row-iterator that scans dirty ranges only, since clean rows hit the cache without needing the run pointers.

- [severity=medium] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:3389-3404`, `:3705-3713`, `:3442-3451` Ś `text_row_slot_order_changed` triggers a **full O(rows) reparent** in `sync_text_resource_nodes::reparent_slots`: every slot is `removeChildNode` + `appendChildNode`, even when only one row's identity moved (294 frames cumulative, 554 ms / mean 1.89 ms; max 43.8 ms). At 235 rows the reparent always touches all wrappers because they're rebuilt as a flat list. Why it matters: this is the dominant per-frame node-graph mutation when a single row scrolls; QSG dirty-flag propagation is paid for every wrapper, not just the moved ones. Concrete follow-up: detect the longest matching prefix and suffix between `old_slots.cache_slots` and `new_slots`, and only move the actually-displaced nodes; alternatively keep the wrappers in a stable order by viewport_row and let the `text_layer` ordering be implicit.

- [severity=medium] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:5804-5830`, `:5975-5976` Ś `build_explicit_snapshot_row_table` partitions `snapshot.cells` into row buckets and runs `std::stable_sort` per row, but cells are already in row-major and column-sorted order at the source (`terminal_screen_model.cpp:2885-2924` appends in `for row { for cell in row_cells }` order; `Terminal_screen_model::publish_render_snapshot` does not reorder them). The pointer-bucketing + sort is therefore guaranteed in-order at construction time and only pays the comparison cost Ś but it still allocates `vector<vector<const Cell*>>(grid.rows)` and walks every cell. Why it matters: O(cells) allocations per packed pass that exist only to recover an ordering the model already guarantees. Concrete follow-up: drop the bucket+sort and iterate `snapshot.cells` linearly using `cell.position.row` to flush a row when it changes; or, again, fold into the merged cell pass and avoid the per-row vector entirely.

- [severity=medium] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:3871-3944`, `:4470-4475` Ś `rect_layer_groups` runs unconditionally per batched-rect layer, including the graphic_rect layer where its self-cost is 1.16 s of `sync_graphic_rect_row_layer`'s 2.35 s (ś49 %). It walks every rect, calls `row_for_rect` (which divides by `cell_metrics.height` and checks rounding tolerance), and builds row-group vectors even when the layer's input key matches the cached descriptor Ś only the *layer* cache-hit check at 4476 comes *after* grouping. Why it matters: ~1.7 ms/frame of grouping work that is thrown away whenever the layer didn't change. Concrete follow-up: shortcut the layer-cache compare to operate on the input content key first (already computed at 6582-6587 for graphics), and only do the row-group decomposition when the input key actually differs.

- [severity=medium] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:3046-3076`, `:3537-3538`, `:4860-4878` Ś Both `clean_cache_skip` and `text_resource_descriptor_reuse` paths invoke `set_row_text_clip_rect` after every clean reuse (cumulative 85,611 invocations). The clip rect rarely changes (only on resize), yet it's compared per row. The compare is cheap (mean ~50 ns), but it implies a per-row write to `cache.clip_rect` whenever Qt's equality fails on float bit-equality. Why it matters: minor, but it's wasted memory traffic on the otherwise-cheap clean path; if a future change makes `QRectF::operator==` more expensive (e.g. tolerance-based) this becomes a hot path footgun. Concrete follow-up: hoist the clip-rect compare outside the loop and only re-set it on `frame.cell_metrics`/`logical_size` changes.

- [severity=medium] [category=architecture] [confidence=medium] `src/qsg_terminal_renderer.cpp:6048-6481` Ś Frame construction allocates 13+ owning `std::vector` fields (`background_rects`, `selection_rects`, `graphic_rects`, `graphic_arcs`, `text_runs`, `cursor_text_runs`, `decorations`, `cursors`, `cursor_graphic_rects`, `cursor_graphic_arcs`, `overlay_rects`, `packed_rows`, `packed_text_spans`, `packed_text_bytes`, `packed_graphic_spans`, `packed_graphic_codepoints`, `dirty_row_ranges`) per frame; many are touched even on clean frames. `text_runs.reserve(cell_count)` allocates room for ~205 k entries per Nelostie frame even when most rows hit the row cache. Why it matters: heap churn on the render-thread hot path and large temporary working sets that may stall on TLB/L2. Concrete follow-up: have `Qsg_terminal_renderer` own a reusable scratch `Terminal_render_frame` and reset it in-place between frames (clear, don't free), or fold frame construction into the renderer so it can size buffers against the dirty subset.

- [severity=medium] [category=production] [confidence=high] `src/qsg_terminal_renderer.cpp:6278-6303`, `:5957-6036` Ś Graphic detection runs **twice**: once in `build_terminal_render_frame::cells` via `append_terminal_graphic_rects` (which mutates `frame.graphic_rects` / `frame.graphic_arcs`) and once in `::packed_data` via the `GRAPHIC_GEOMETRY` classification path that emits `packed_graphic_*`. Both paths are needed today because `graphic_rect_layer_inputs` (2157-2184) prefers `packed_rows` when available and falls back to `frame.graphic_rects` otherwise. Why it matters: cells that contain box-drawing/block codepoints incur two classifier evaluations plus two emission paths. Concrete follow-up: collapse to a single emit site (the packed path) and rebuild `frame.graphic_rects` only as a thin view over the packed graphic spans, or remove the legacy `frame.graphic_rects` path entirely if no test depends on its codepoint coverage outside the packed set.

- [severity=low] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:3096-3121`, `:3493` Ś `text_frame_cache_key` is built fresh per frame (1,406 ns mean, 930 Ąs cumulative) even though it only changes on font / `logical_size` / `cell_metrics` / `grid_size` / `active_buffer` transitions. Hot but small; the same effective comparison can be done with two scalar checks plus a font generation counter. Why it matters: tiny but illustrates duplicated work that scales with the number of cache layers. Concrete follow-up: stash the last-frame key fingerprint as a small struct of scalars and only rebuild the QByteArray on miss.

- [severity=low] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:2070-2082`, `:6629-6643` Ś `rects_for_frame_group` and `graphic_rect_layer_inputs` allocate fresh `std::vector<Terminal_render_rect>` copies on every paint call (38,298 cumulative `row_local_rects` calls within `sync_graphic_rect_row_layer`, 560 ms total). Concrete follow-up: pass spans/views through the layer-sync pipeline rather than owned vectors, or reuse a scratch buffer attached to the cache slot.

- [severity=low] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:1006-1042`, `:3252-3258`, `:3929-3936` Ś `row_cache_identity_t` for geometry layers omits `retained_line_id` and `content_generation` (geometry slots compare only `active_buffer` + `logical_row`), while text slots compare all four. Concrete follow-up: add a comment block at `row_cache_identity_t` documenting the two-tier scheme, or unify on the four-field identity if it doesn't change behavior.

- [severity=low] [category=consolidation] [confidence=medium] `src/qsg_terminal_renderer.cpp:1095-1123`, `:3186-3213` Ś `Text_resource_row_descriptor` and `text_resource_key` represent the same content twice: the descriptor is a structured per-run record with QString text, while the key is a byte serialization that includes the QString UTF-16 payload. With the descriptor-reuse path already covering 1,758 successful reuses and the key never matching, the descriptor is the load-bearing representation and the key is dead weight. Why it matters: two overlapping fingerprints make the code harder to reason about and double the cache-miss cost. Concrete follow-up: pick one (descriptor) and delete the other in the dirty-rebuild branch; if a byte-level fingerprint is desired, hash the descriptor instead of serializing it.

Concrete follow-up: add either a test that asserts `text_key_match_reuses > 0` for some scenario, or Ś if the key is intentionally removed Ś a regression test asserting the descriptor path catches the scenarios previously expected to hit the key.

- [severity=low] [category=missing-coverage] [confidence=high] No test asserts the scaling property of `text_wrapper_order_rebuilds` vs grid size. A one-row scroll on a 235-row Nelostie grid currently rebuilds all 235 wrapper attachments; a test that asserts reparent count = number of actually-displaced rows would catch the O(rows) regression. Why it matters: this is the kind of issue that only shows up in profiles, not functional tests. Concrete follow-up: add a benchmark-style test (or a stat-assertion test) on `sync_text_resource_nodes` that bounds wrapper reparent work to the changed-identity count.

- [severity=low] [category=missing-coverage] [confidence=medium] No test asserts cross-pass classification parity between `build_terminal_render_frame::cells` and `::packed_data`. Both paths separately call `classify_terminal_simple_content_cell` and `snapshot_row_is_dirty`; a future change that mutates one classifier without the other would silently produce a frame where `text_runs` says "text" and `packed_*` says "graphic" (or vice versa). Why it matters: with two parallel emission pipelines, drift is a real risk Ś and the unification suggested above will be hard to land safely without parity coverage. Concrete follow-up: add a property test that asserts each cell is represented in exactly one of (`text_runs`, `packed_text_spans`, `packed_graphic_spans`, cursor-overlap, ime-overlap), or that `text_cells_rendered_as_text + text_cells_rendered_as_graphic + text_cells_empty + cursor_text_runs.size() + ime overlay == cells_rendered`.

- [severity=low] [category=test-quality] [confidence=medium] `tests/qsg_render/qsg_render_tests.cpp:1764-1770` (and 3843 onwards) clears `packed_text_spans` / `packed_text_bytes` to prove they don't affect visual output. This is good as a pixel-parity assertion but acts as a *de facto* contract that the renderer must keep ignoring them Ś which conflicts with any future plan to consume the packed text bytes. Why it matters: the test, as written, will block the natural follow-up of "use packed bytes from the renderer." Concrete follow-up: split this test into (a) "packed sidecars produce the expected stats" and (b) "rendering the same content with and without packed sidecars yields equivalent pixels" *only if* the renderer's contract is that they remain sidecars indefinitely.

## Test Assessment

The QSG-render scope is well exercised at the unit level. `tests/qsg_render/qsg_render_tests.cpp` has ~60 named test functions covering: text resource descriptor reuse across decoration / hyperlink / styled-ASCII / preedit / clipped-run boundaries; row-identity scroll and resize cleanup; batched-rect cache for background/selection/decoration/graphic layers; software/AA fallback transitions; cursor and IME overlay paths; lifecycle/teardown; packed sidecar pixel parity. `tests/render_frame/render_frame_tests.cpp` covers frame stats classification, packed-text/graphic order, cursor/IME exclusion from packed text, and viewport-empty handling. The unit coverage of the row identity + descriptor logic is unusually thorough.

The main blind spots are the *scaling* properties: nothing tests that the renderer scales sublinearly with grid size in the common dirty-subset case, so the O(rows) reparent and the duplicate dirty-row scans pass CI without resistance. Nothing tests that the dead `text_resource_key` path is reachable or unreachable Ś both refactor directions are silent in CI. Nothing tests that the two classifier passes (`::cells` and `::packed_data`) agree, so the consolidation suggested above carries silent-skew risk. The packed-sidecars-do-not-affect-visual-output test (qsg_render_tests:3843) is correct as a pixel-parity assertion but conflates "sidecars exist" with "sidecars are not consumed" Ś splitting those would make the test more refactor-friendly.

## Suggested Follow-ups

1. **Eliminate the packed-text-bytes emission unless a consumer exists.** Confirm via grep + test that `frame.packed_text_spans` / `packed_text_bytes` have no production reader; if so, remove them from `build_terminal_render_frame_packed_data` and reclaim ~6 ms/frame. Lower the test contract per the last finding.
2. **Merge `build_terminal_render_frame_packed_data` into the cells pass.** Drive emission off the classifier result already computed at `:6141`. Validate by adding cross-pass parity coverage first, then assert the combined `build_terminal_render_frame` time falls toward the larger of the two current totals.
3. **Drop `text_resource_key` from the dirty-rebuild path** (or instrument and demonstrate a scenario where descriptor coverage is insufficient). Either way, kill the 100 ms of unused key work and the byte-serialization heap traffic.
4. **Stop eagerly building `text_resource_row_descriptor`** when the group is dirty and the clean-cache-skip preconditions are not met. Replace with a cheap predicate for the clean-skip gate (already 90 % of considered groups).
5. **Cache `snapshot_row_is_dirty` as a `vector<bool>` once per frame** before the cell pass. This is a prerequisite for the cells/packed merge anyway.
6. **Refactor `sync_text_resource_nodes::reparent_slots`** to identify and move only the actually-displaced wrappers (prefix/suffix match against `old_slots.cache_slots`), so a single-row scroll touches O(displaced) nodes instead of O(rows).
7. **Add a scaling test** that bounds `text_wrapper_order_rebuilds` cost and `frame_dirty_row_lookup_count` per Nelostie-style frame so regressions in the consolidation work are caught by CI.
