## Findings

- **[severity=high] [category=production] [confidence=high]** `src/terminal_screen_model.cpp:2885` (and the committed equivalent at `:2766` of HEAD) — `render_snapshot::append_rows` loops every viewport row unconditionally and emits ~89.5 K `Terminal_render_cell` push_backs per snapshot through `append_snapshot_cells_from_row` (`:7076-7100`). At 1498 snapshots this is 134 M cell-pushes and accounts for **66% of gui-thread wall time**. The renderer already gates per-row work on dirty status (`sync_text_resource_nodes::clean_cache_skip` at `qsg_terminal_renderer.cpp:3535`) and identity (`row_cache_identity_has_valid_retained_provenance`), so the cell payload for clean rows is wasted producer-side work that the consumer discards. Concrete follow-up: prototype a delta snapshot contract where `snapshot.cells` only carries dirty rows and `visible_line_provenance` carries identity for clean rows; A/B against `nelostie_profile_scalar_span` and a stable-content sibling profile to bound the win.
- **[severity=high] [category=production] [confidence=high]** `src/terminal_session.cpp:5969`, `:5994-6010` — `publish_render_snapshot` always calls `m_screen_model->render_snapshot(request)` *first*, then applies `snapshot_with_coalesced_dirty_rows` (`:879`) which only edits `dirty_row_ranges`. When `m_render_snapshot_synced_generation < m_render_snapshot_generation` (renderer hasn't consumed prior snapshot), the heavy snapshot is built and immediately superseded. Profile: 836 snapshots painted-then-discarded × ~7 ms each ˜ **5.85 s wasted out of 16 s**. Concrete follow-up: gate the call to `m_screen_model->render_snapshot()` on "renderer caught up" or build a cheap "dirty-only patch" then fold into the still-unread snapshot. The infrastructure (`m_render_snapshot_synced_generation`) already exists — only the early-exit is missing.
- **[severity=high] [category=architecture] [confidence=medium]** Backend-output cadence drives publication 1:1 (`terminal_session.cpp:4376-4394`). 1500 segments ? 1498 publications. Snapshot publication is throttle-free; nothing downstream of the parser tries to merge consecutive segments. Why it matters: this is the upstream cause of #2 — even a perfect publish-gate still pays parser/ingest time per segment. Concrete follow-up: instrument segment-merge feasibility by adding a `BACKEND_OUTPUT` coalesce step in `process_pending_commands` (`:3094-3159`) when adjacent same-kind commands sit in the queue; measure how many of the 1500 segments arrive in batches and could be combined before any model work.
- **[severity=medium] [category=production] [confidence=high]** `src/qsg_terminal_renderer.cpp:6135` — renderer cell-pass also iterates every cell in `snapshot->cells` (cumulative `frame_cell_pass_input_cells=55.8 M`, `frame_dirty_row_lookup_count=111.7 M` ˜ 2× cell count). Each iteration calls `snapshot_row_is_dirty` (`:5793-5802`) which linearly scans `dirty_row_ranges` per cell. This is the second-largest implicit dependence on the "full" cells layout — a delta snapshot (Finding #1) would also halve cell-pass work. ## Test Assessment

- **No model-side benchmark for `render_snapshot::append_rows`.** `benchmarks/embedded_terminal/embedded_terminal_benchmark.cpp` hand-crafts snapshots and feeds them to the renderer — it never exercises `Terminal_screen_model::render_snapshot()`. The Phase 8 ring benchmark covers retained-history materialization (12.6 µs/row) but is unrelated to the full-screen snapshot loop. **Recommend:** add a synthetic ingest harness that drives N rows × M dirty rows through `Terminal_screen_model` and reports `append_rows` ns/snapshot per dirty-row count.
- **`tests/render_snapshot/render_snapshot_tests.cpp` is 1022 LoC and never asserts dirty-row counts vs cell counts.** A regression that shipped clean-row cells in the snapshot would not be caught here; the profile counters (`render_snapshot_dirty_rows_visible`, `render_snapshot_cells_emitted`) exist but no test asserts the ratio.
- **`tests/backend_session/backend_session_tests.cpp` (13.3 K LoC)** covers ordering, lifecycle, and snapshot publication, but I did not find a case asserting "publish-gate": if the renderer hasn't consumed snapshot N-1, publishing N should reuse without rebuilding cells. The semantics are not yet codified, which is consistent with the absence of an early-exit in `publish_render_snapshot`.
- **`tests/screen_operations/model_ops_tests.cpp` (3532 LoC)** exercises mutation/dirty interaction but does not measure end-to-end cells emitted per dirty-row count. A test like *"after `mark_dirty(7)` followed by `render_snapshot`, `cells` contains only row 7's data"* would lock in the delta-snapshot direction.
- **`sync_text_resource_nodes` (qsg_terminal_renderer) is exercised by the embedded benchmark** but the benchmark feeds full-cell snapshots, so it does not pressure the cell-pass with realistic dirty-row distributions. The 84.3 K cells/frame in the cumulative renderer stats is exactly what's pushed through; a benchmark with dirty-only snapshots would reveal the renderer-side savings.
