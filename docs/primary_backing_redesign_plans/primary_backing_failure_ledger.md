`primary_backing_buffer_consolidated_design.md`. It records motivating scenarios
before behavior changes are made. Later phases may add entries or replace
`not yet covered` rows with exact tests, but they must update this file and name
the reason when changing a classification.

## Ledger maintenance rules

1. Do not reclassify a scenario without recording the new evidence and owning
   phase.
2. Do not use recovery-enabled success as proof for core storage, viewport,
   resize, selection, or publication semantics.
3. Promote `not yet covered` entries to exact tests before changing production
   behavior for their owning phase.
4. Keep recovered-row behavior out of this ledger until deferred Phase R
   recovery-policy work.
