# Review

Session: `02_claude_scaling_strategy` — Large-grid dirty-row and rendering scaling strategy investigation
Profile: `vnm_terminal/build/profiles/nelostie_profile.txt` (233 rows × 871 cols, ~107s wall, GUI thread 48.3s, render thread 19.7s)

## Findings

- **[severity=high] [category=architecture] [confidence=high] `src/terminal_screen_model.cpp:4524-4535,4181-4197,4154-4179` — Every ASCII span deep-copies the row (`before_cells = screen_row.cells`, ~42 KB at 871 cols) and re-scans the whole row element-by-element (`rows_have_same_selection_content` over 4 cell fields including `QString`) just to gate `++row.retained_line_provenance.content_generation`.** This is per-span (`put_printable_ascii_text` slices into spans bounded by autowrap/`available_columns`), so on the dominant ASCII fast path the model pays O(columns) work twice (copy + compare) per span on top of the per-char `clear_cell_at`/six-field write. Measured per-call mean 74.9 µs on `print_text` with 472K calls and 7.87M character marks aligns directly with this pattern. **Why it matters.** This is the single largest leverage point in the run (~35 s / 73% of gui_thread). **Follow-up.** Replace before/after diffing with an inline "row mutated" boolean set inside `write_printable_ascii_cell_content` (and the few other writers that legitimately need generation bumps); only bump `content_generation` when the boolean is set. Drop the row copy entirely. Expect 4-8× reduction in per-span cost on ASCII-heavy workloads.

- **[severity=high] [category=architecture] [confidence=high] `src/terminal_screen_model.cpp:2850-2890,6770-6794,6850-6877` — `render_snapshot::append_rows` walks all `m_config.grid_size.rows` (233) unconditionally and, per row, copies the entire `std::vector<Cell>` from the backing buffer via `viewport_row_cells()` returning `std::optional<std::vector<Cell>>(backing->cells)` (lines 6864, 6876), then pushes every occupied cell into a flat `snapshot.cells` (max 102,399 entries per snapshot), then re-scans the just-pushed range for hyperlinks.** `request.dirty_rows` is already computed and compacted (lines 2844-2847), but is used only as metadata — it does NOT prune which rows get copied. The 6.23 ms/call mean × 1,252 calls = 7.8 s. **Why it matters.** A snapshot of only the dirty rows would shrink this cost roughly proportionally to dirty fraction; the consumer (`terminal_public_projection.cpp:329-417`) currently asserts complete coverage so the publisher must always be complete. **Follow-up.** Introduce per-row generation in the snapshot (`snapshot.row_revisions[row] = content_generation`) and have the consumer keep a parallel cache keyed by `(retained_line_id, content_generation)`. When the snapshot says revision is unchanged, the consumer reuses its existing per-row cells without the publisher having to ship them. This is the textbook "incremental snapshot with generation tags" pattern and converts append_rows from O(rows × cols) into O(dirty_rows × cols).

- **[severity=high] [category=architecture] [confidence=high] `src/qsg_terminal_renderer.cpp:3491-3500,3493-3543,3626-3678,3687-3695` — `sync_text_resource_nodes` keys cache reuse on `current_text_frame_key == root.text_frame_key`. `text_frame_cache_key` hashes the *entire frame*, so any dirty row mutates the frame key and `same_text_frame_key=false` makes the three reuse fast paths (`clean_cache_skip`, `text_resource_descriptor_reuse`, `key_match_reuse`) all unreachable; every group is rebuilt, every old node is `delete`d (recursive `delete_node_tree` ~65 µs per entry), every wrapper is reparented (`reparent_slots` O(n²) sibling traversal, 1.8 ms/call mean × 284 calls).** Last frame shows the worst-case manifestation: 163 dirty groups, 0 reused, 99,189 child nodes cleared. **Why it matters.** This makes the text path's per-frame cost O(visible rows) instead of O(dirty rows) — the same pathology as append_rows, but with QSG-node allocation/destruction overhead amplifying it. It is the dominant render-thread cost (10.5 s / 53%). **Follow-up.** Drop `text_frame_key` as a global reuse gate, or split it into a "geometry/font/style-palette" key (stable across frames) and per-row content keys (compared in the per-group loop you already have). The per-group `clean_cache_skip` path already has the right shape — let it be reachable when only some rows are dirty. Add a metric `text_groups_skipped_unchanged` and assert in tests that scrolling 1 row in a 233-row frame yields ≥ 230 skips.

- **[severity=high] [category=architecture] [confidence=high] `src/qsg_terminal_renderer.cpp:2901,3603-3625,~969` (Terminal_text_resource_node destructor) — `make_text_resource_node` always `new`s a fresh `Terminal_text_resource_node` and the replace path `delete`s the old one; there is no in-place node reuse even when the new content fits in the same geometry/leaf-node shape.** `delete_node_tree` walks the subtree linearly using `removeChildNode` (potentially O(n) per sibling under Qt's intrusive list). With 15,336 `replace_cache_entry` calls and `text_cache_entry_max_child_nodes_cleared=667`, this is amortized but persistent overhead. **Why it matters.** Allocation/destruction churn dominates the 988 ms in `replace_cache_entry` and 434 ms in `remove_stale_entries`. **Follow-up.** After the frame-key fix above, the next step is in-place leaf updates: when a row's coalesced text matches the prior key shape (same run count, same fonts, same geometry width), update existing `QSGGeometryNode`s and re-issue glyph runs without destroying the QSG subtree. This is consistent with Qt Quick's `updatePaintNode` contract and turns row updates into the cheap path (geometry update + material change).

- **[severity=medium] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:6080-6293` and `:5920-6000` — `build_terminal_render_frame` makes two full passes over `snapshot->cells` (102,399 cells/frame): `::cells` builds `text_runs`/`decorations`/`graphic_rects`, then `::packed_data` revisits the same cells for the packed text/glyph descriptors.** Combined 8.3 s / 28 ms per frame is roughly half of `updatePaintNode`. **Why it matters.** Two passes double cache misses across what is the same snapshot vector, and the first pass already touches every field the second needs. **Follow-up.** Fuse the loops; emit packed text descriptors as a fall-out of the same classification pass instead of re-walking. Less ambitious win than the snapshot/QSG changes but the structural change is small and lets the dirty-row pruning above also shrink this loop.

- **[severity=medium] [category=architecture] [confidence=high] `src/terminal_screen_model.cpp:1942-1958, 6292-6329` — `publish_pending_changes` runs on every `apply_action` (991,674 calls / 1,285 ingest batches ≈ 770 publishes per ingest), and inside it does `publication.dirty_rows.insert(m_dirty_rows.begin(), m_dirty_rows.end())` against `std::set<int>` on both sides.** Aggregate cost is only 106 ms today, but the design pattern is "publish on every action even if nothing changed" — `published_unique_rows=53,632` over 991K calls is < 0.06 rows/publish. **Why it matters.** Currently a low-cost no-op majority of the time, but every future per-row metadata addition to publication compounds onto each of those 991K calls. The mark→publish→finalize chain also uses three different containers (`std::set` → `std::set` → `std::vector`) with implicit copies. **Follow-up.** Publish once at the end of an `ingest()` batch (or on synchronized-output release) rather than per action; the per-action publication only matters when an external consumer is observing mid-batch, which the current consumer is not. Optionally replace `m_dirty_rows`'s `std::set<int>` with `std::vector<int>` + `m_dirty_row_seen_bitset` (272-bit fixed-size bitset for max grid) to make the per-mark cost truly O(1) without log-n tree work.

- **[severity=medium] [category=production] [confidence=medium] `src/qsg_terminal_renderer.cpp:2507-2580 (prepare_text_layout), :2793-2836 (append_batched_text_run_nodes)` — `prepare_text_layout` and `add_text_run_layout` run unconditionally per `Terminal_render_text_run` (516,835 calls each), with no run-content deduplication across rows or frames.** The mean (4.3 + 7.5 µs) is dominated by Qt's `QTextLayout` shaping, which is not free even for ASCII. With coalescing already shrinking 12.66 M raw runs → 516 K resource runs (24×), the residual is still per-run shaping. **Why it matters.** Combined 6.1 s — third-largest render item. **Follow-up.** Add a frame-local (or short-lived LRU) cache keyed by `(text, font, foreground_metrics)` for `QGlyphRun` outputs. For long-lived terminals the working set of distinct short runs is small (commands, prompts) and would hit the cache hard.

- **[severity=medium] [category=production] [confidence=high] `include/vnm_terminal/internal/terminal_screen_model.h:389-397` — `Cell { QString text; int display_width; bool wide_continuation; bool occupied; Terminal_style_id style_id; std::uint64_t hyperlink_id; }` is at least ~48 bytes plus a `QString` (24 bytes + heap allocation for non-inline payload).** A 233 × 871 grid is ~203K cells ≈ 10 MB+ plus QString heap allocations. Every row copy in print_text and snapshot append touches both the inline payload and the QString refcount bumps. **Why it matters.** It's the silent multiplier on every "copy a row" path. Even after the diff/snapshot fixes above, each remaining copy still drags this footprint. **Follow-up.** For ASCII-only cells (the 80% case per cumulative stats: `text_cells_printable_ascii=14.02 M / text_cells_rendered_as_text=14.73 M`), store the codepoint inline (uint32 or small-string) instead of allocating a QString per cell. Defer to a heap representation only for grapheme clusters of size > 1.

- **[severity=low] [category=production] [confidence=high] `src/terminal_screen_model.cpp:6299` — On every publish, `publication.dirty_rows` (a `std::set<int>`, header `terminal_screen_model.h:534`) absorbs `m_dirty_rows` via range-insert. The downstream `Terminal_screen_model_result::dirty_rows` (header line 173) is a `std::vector<int>`, requiring another conversion in `finalize_ingest_result`.** Three containers for what is logically "the set of dirty row indices, bounded by ≤ rows count." **Why it matters.** Container plurality slows the path and makes scope creep easy. **Follow-up.** Unify on a small `Dirty_row_set` value type backed by a bitset (max 4096 rows ≈ 512 bytes) plus an `extract_sorted()` accessor. Removes set/vector copy cost and obviates the duplicate-mark check (bit OR is idempotent).

- **[severity=low] [category=consolidation] [confidence=medium] `src/terminal_screen_model.cpp:2856-2889` — The append-rows loop has three sibling helpers (`viewport_row_cells`, `viewport_row_retained_hyperlink_identity_keys`, `viewport_row_provenance`) each performing its own viewport→backing translation per row.** **Why it matters.** Repeated address translation work per row × 233 × 1,252 calls is small but multiplied by the snapshot-rebuild pathology above. **Follow-up.** Fold these into one viewport-iterator that yields `{cells&, hyperlinks*, provenance*}` per row to halve the per-row index work. Defer until after the dirty-row pruning win above (which is the bigger lever).

- **[severity=low] [category=production] [confidence=high] retained-history ring (`src/terminal_history_row_record_codec.cpp`, `src/terminal_history_ring.cpp`) — Confirmed not material at this stress level: ~15 ms across the whole run, no hidden O(scrollback) walks, `make_room_for` is O(records_evicted), `commit` is O(1).** Encoding cost (`prepare_encode_parts` 12 µs/row, `write_payload::cells` 3.8 µs/row at 871 cols ≈ 14 ns/cell) is proportional to row width and small. **Why it matters.** Eliminating the redundancy of `prepare_encode_parts` (12 µs/row × 458 rows = ~5.5 ms total) is not worth the validation-loss tradeoff. **Follow-up.** No action needed. Keep an eye on `m_records.push_back()` on the deque if scrollback grows unbounded; the deque resize policy in libstdc++ is segmented so it's safe, but document that.

## Test Assessment

Strength of current tests for this scope is **poor for scaling**. Functional correctness coverage is solid (`tests/render_snapshot/render_snapshot_tests.cpp` has 14 tests, `tests/screen_operations/model_ops_tests.cpp` 26, `tests/qsg_render/qsg_render_tests.cpp` 65), but every meaningful test grid is small (≤ 9 × 13 in model_ops, ≤ 12 cols in render_snapshot, ≤ 5 visible rows in qsg_render). None of the production-relevant scalability invariants is asserted:

- **No test exercises a 233-row dirty grid through `render_snapshot::append_rows`** — the path that costs 7.8 s in the profile has no test that would observe a regression there.
- **No test verifies that text-cache reuse actually fires for partial invalidation at scale.** `tests/qsg_render/qsg_render_tests.cpp:5970-6040` does check `text_content_reused == 2` for a tiny scenario, but the frame-wide cache-key invalidation (the architecture finding above) would still pass these tests because the test scenarios never compare a frame that should reuse most rows.
- **Dirty-row statistics (`mark_requests`, `duplicate_mark_requests`, `published_unique_rows`) are not asserted anywhere.** The profiler counters exist but no test pins them.
- **The history ring is the only path with a real benchmark** (`history_row_record_phase8_benchmark.cpp`, 2000 rows). That's the path that turned out to be immaterial — the actual hot paths are untested at scale.
- **Misleading test signal:** the heavy presence of small-grid coverage creates false confidence that the design "works at any size." It works correctly; it does not work efficiently.

## Suggested Follow-ups

1. **Add scaling-regression tests first** (validation gate, before any fix): a 233×871 fixture that streams an ASCII screenful, captures `print_text` time, `append_rows` time, and `sync_text_resource_nodes::text_content_reused/text_groups_clean`. Pin counters for: marks-per-char, snapshot rows-copied-per-snapshot, text-groups-reused-per-frame on the partial-dirty case.
2. **Phase 1 — kill the row-diff** (finding 1): inline mutation flag in `write_printable_ascii_*` and friends; remove `before_cells` copy and `rows_have_same_selection_content` from the print path. Re-profile.
3. **Phase 2 — incremental snapshot** (finding 2): per-row generation token in snapshot, consumer caches by `(retained_line_id, content_generation)`. Drop the "complete coverage" requirement in `terminal_public_projection.cpp`.
4. **Phase 3 — per-row text cache key** (findings 3, 4): replace the frame-wide `text_frame_key` with stable-context + per-row content keys, so partial-dirty frames hit the existing reuse paths.
5. **Phase 4 — in-place QSG leaf updates and shaping cache** (findings 4, 6): only after phases 1–3 land, since they may reshape the working set.
6. **Phase 5 — Cell layout** (finding 7): ASCII inline storage. Touches every row copy site so easier after the diff is gone.
7. Each phase: validation = the new scaling tests must show the expected counter improvements *and* existing functional tests must remain green. Keep the perf delta visible in `last_renderer_stats` so future regressions are caught.
